Problem

The Red Caps provide value to the community from 6th to 11th and from Lincoln to Pearl Street. Their current data collection process is handwritten notes on a scratch pad, along with data entry on log sheets in the office at the end of each shift. Since all data is on paper, it limits the opportunity for the information to be shared with our community and evaluated by stakeholders. A more efficient data collection process would enable the Red Caps to spend more time out and about in the Downtown District. It would also provide more data to drive decisions by policy makers.

Challenge

Give some love to the Red Caps by way of technology that can enable easy and efficient collection of data through a mobile interface.The data collected: way-finding, meet and greets, escorting employees to their vehicles after dark, responding to emergency calls, reports of vandalism, locking and unlocking buildings for property and business owners and security checks. Then enable a reporting interface or data export mechanism so that the data can be used for the benefit of our community, to identify trends, and to also shine a light on the hard work of the Red Caps.

Infrastructure

While the team built a system that is easily portable, being built on the back of Wordpress, we still wanted to offer a robust system. We have therefore built the system on Amazon Web Services - this is a cloud native solution.

We wanted to be able to just hand over they keys at the end of the day, so we needed to set up an AWS account just for the Red Caps. That also meant having an email address with which to register and use for the past 40-something hours. Then AWS, despite the fact that our offering will run in the free tier, required a credit card. Again, wanting to be able to just turn everything over, we put a Visa gift card on the account.

We're turning over the following assets to the Red Caps:

​

  • The i.heart.red.caps@gmail.com email address
  • An AWS account registered under that email address
  • A $20 gift card

This setup should run without further cost for at least a year. At which point it should cost approximately $10/month, but could be reduced to $50/year with the pre-purchase of resources.

C.A.R.D.S.

Capacity

As an educated guess, and barring bad actors, they should have ample Capacity for the foreseeable future. The AWS server runs WordPress, the API that ties the app to WordPress, and the Disaster Recovery systems.

Availability & Redudancy

At this scale, Availability and Redundancy are effectively the same, which is to say, there's little capacity for either without significant increases in running costs. For the most part we're reliant on Disaster Recovery but the entire system, as delivered, has been baked into a Machine Image (AMI). So if the server shuts down, it can be restored to today’s state in a matter of minutes.

Disaster Recovery

Data can never be recreated so we went the extra mile in terms of Disaster Recovery. There are two levels of backups happening. First we do a full dump of the MySQL database containing all the data (incidents and Wordpress) on a daily basis. Additionally, I put a system in place that creates a clone of the server's hard drive every day.

The MySQL backups are are kept via S3 versioning for 30 days, put in a cheaper but less redundant storage system for 60 days, then finally deleted after a total of 90 days to save total running costs
